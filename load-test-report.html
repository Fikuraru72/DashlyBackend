01 / Executive verdict

Pipeline works.
Capacity is not yet proven.

Dashly berhasil melewati validasi produksi dari MQTT sampai dashboard Socket.IO, dengan nol error pada run bersih satu peserta. HTTP juga kuat pada dua skenario. Namun, hasil 100 peserta menunjukkan backlog serius dan dijalankan sebelum seluruh prasyarat simulator benar. Kesimpulan yang jujur: sistem operasional, tetapi belum layak dinyatakan siap untuk beban balapan 100 peserta secara berkelanjutan.

04 / Invalid & diagnostic runs

What failed—and what
it actually means.

  1. 18:38 UTC
    Broken MQTT route

    100-participant run produced 0 publishes and 861 errors. SSH forwarded to a listener that was not reachable from VPS loopback. Result measures a broken tunnel only.

  2. 19:00 UTC
    Stale local forward

    Port 1884 remained owned by an old SSH process. A second tunnel could not bind. Runner continued instead of failing fast.

  3. 19:02 UTC
    Polling without cookie jar

    Traefik correctly issued dashly_api, but Node's Socket.IO simulator lacked withCredentials. Polling requests crossed replicas and failed.

  4. 19:04 UTC
    Participants not TRACKING

    Worker logs explicitly showed state=REGISTERED, dropping. Production behavior was correct; preparation was incomplete.

  5. 19:10 UTC
    Clean E2E pass

    After tunnel cleanup, sticky credentials, and TRACKING activation: 14/14 positions delivered, zero transport errors, p95 402 ms.

Do not cite as capacity

OSRM tunnel run

  • 100% timeout
  • OSRM CPU stayed at 0%
  • No bytes received

The request path never exercised routing compute. Run OSRM directly from the VPS network or a controlled external endpoint.

Use as warning only

100-participant race run

  • MQTT accepted 11,292 publishes
  • Only 6,540 socket samples across 2 sockets
  • E2E p95 climbed to 76.64 seconds

The monotonic latency rise is the signature of a queue draining slower than input. But invalid participant state prevents a clean capacity claim.

06 / Engineering analysis

The likely constraint
is downstream compute.

MQTT ingest is not the observed bottleneck. Worker throughput is.

During the 100-participant diagnostic, the local MQTT clients acknowledged all 11,292 publishes without error, while Socket.IO output lagged progressively. At the later infrastructure snapshot, both workers were CPU-heavy while ingest was nearly idle. Together, these observations point toward enrichment, analysis, ranking, persistence, or fan-out work after ingestion.

Not yet isolated.
There is no queue-depth and job-lag time series attached to these runs. The report therefore identifies a likely subsystem, not a proven function or query.

Two facts prevent stronger attribution: the 100-participant dataset was not fully normalized to TRACKING before that run, and worker CPU was captured as a later snapshot rather than throughout the test. The next run must capture queue wait, active/waiting/failed counts, worker CPU, Redis latency, and database latency on the same clock.

Capacity claim permitted today

Dashly handles smoke-level production traffic and a single active participant with sub-second realtime delivery. No defensible maximum participant count has been established.

Security note

An access token was exposed in terminal/chat output. Do not place it in this report or source control. It expires quickly; obtain a fresh token for every test. If long-lived revocation is required, add token versioning or a denylist only when operational policy demands it.

07 / Next controlled test

Earn the race-day sign-off.

Do not jump directly to a long 100-participant run. Establish the knee of the curve with short, reproducible stages, using the corrected preparation and simulator.

  1. Re-run the corrected 2-minute smoke

    10 active participants, 10 spectators, 2 operators, 2 sockets. Require publish errors 0, socket errors 0, HTTP failure below 1%, and E2E p95 below 3 seconds.

  2. Step participant load: 25 → 50 → 75 → 100

    Five minutes each, same 1-second GPS interval. Stop when E2E latency grows continuously, queue depth does not recover, errors exceed 1%, or containers restart.

  3. Record synchronized telemetry

    Capture host CPU/RAM/swap, per-container CPU, BullMQ waiting/active/failed counts, oldest-job age, Redis latency, Supabase query latency, and restart count every 5 seconds.

  4. Separate routing capacity

    Test OSRM directly on the Docker/VPS network. The SSH-tunnel timeout run is discarded and must not appear in capacity planning.

  5. Fix Mosquitto health semantics

    Replace the health check that waits for a $SYS message with a deterministic broker connectivity check compatible with the current Mosquitto configuration.

  6. Only then run soak

    After 100 participants meet SLO for 30 minutes, run 3–6 hours with realistic reconnects and offline sync. Verify queues return to baseline after bursts.

08 / Method & provenance

What was inspected.

HTTP evidence. k6 summary logs for health stress and real-world spectator/operator paths. Thresholds are the suite's declared pass criteria.

Realtime evidence. Node simulator metrics, participant state endpoint responses, worker logs, MQTT tunnel connectivity, Socket.IO sticky cookie, and polling behavior.

Infrastructure evidence. Live docker ps, docker stats --no-stream, restart counters, host uptime, memory, swap, and load average.

Classification rule. A run is “valid” only if its generator and all intended protocol paths were operational. A run with broken forwarding, missing affinity credentials, or invalid domain state is diagnostic—not capacity evidence.
